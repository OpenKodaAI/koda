# Architecture Overview

For the public product-facing version of this material, start with [`../README.md`](../README.md), [`../architecture/overview.md`](../architecture/overview.md), and [`../architecture/runtime.md`](../architecture/runtime.md). This document remains focused on repository orientation for contributors and LLM tooling.

This repository implements a Telegram-native engineering assistant that delegates code work to provider CLIs, currently Claude Code CLI and Codex CLI, and augments them with repository tools, external integrations, and memory.

For the canonical machine-readable file, flow, and test index, start with [`repo-map.yaml`](repo-map.yaml). It is generated by [`../../scripts/generate_repo_map.py`](../../scripts/generate_repo_map.py). For provider entrypoints and compatibility rules, read [`llm-compatibility.md`](llm-compatibility.md).

## Entry Points

- [`../../koda/__main__.py`](../../koda/__main__.py): main application startup and handler registration
- [`../../agent.py`](../../agent.py): compatibility wrapper that imports and runs `koda.__main__.main`
- [`../../launcher.py`](../../launcher.py): starts multiple named agents as separate processes
- [`../../docker-compose.yml`](../../docker-compose.yml): container runtime definition

## Runtime Layers

### Telegram Interface

- [`../../koda/handlers`](../../koda/handlers) receives Telegram commands, text messages, documents, images, audio, and callback interactions.
- Handlers authenticate the user, normalize Telegram input, track provider/model preferences, and pass work into the service layer.

### Orchestration Layer

- [`../../koda/services/queue_manager.py`](../../koda/services/queue_manager.py) is the central orchestration module.
- It prepares the effective compiled system prompt, creates or restores the canonical agent session, enqueues work per user, runs the selected provider in streaming mode, loops through runtime agent tools when needed, applies provider fallback when necessary, and sends the final response back to Telegram.

### LLM Execution Layer

- [`../../koda/services/llm_runner.py`](../../koda/services/llm_runner.py) is the provider-neutral execution facade used by the orchestration layer.
- [`../../koda/services/claude_runner.py`](../../koda/services/claude_runner.py) wraps the `claude` CLI process.
- [`../../koda/services/codex_runner.py`](../../koda/services/codex_runner.py) wraps the `codex` CLI process.
- The runner layer handles command construction, streaming and non-streaming execution, retryable errors, native event parsing, usage and cost capture, and provider-native session resume data.

### Runtime Tool Layer

- [`../../koda/services/tool_prompt.py`](../../koda/services/tool_prompt.py) advertises agent-specific tools inside the system prompt.
- [`../../koda/services/tool_dispatcher.py`](../../koda/services/tool_dispatcher.py) parses `<agent_cmd>` tags, enforces read/write rules, and executes the matching tool handlers.
- Focused integration services such as [`../../koda/services/browser_manager.py`](../../koda/services/browser_manager.py), [`../../koda/services/db_manager.py`](../../koda/services/db_manager.py), [`../../koda/services/atlassian_client.py`](../../koda/services/atlassian_client.py), and [`../../koda/services/http_client.py`](../../koda/services/http_client.py) provide the actual capability surfaces.

### Artifact And Deep-Context Layer

- [`../../koda/services/artifact_ingestion.py`](../../koda/services/artifact_ingestion.py) is the provider-neutral artifact pipeline for local files, URLs, OCR, document parsing, spreadsheet extraction, audio transcription, and video frame analysis.
- [`../../koda/services/jira_issue_context.py`](../../koda/services/jira_issue_context.py) builds proactive Jira issue dossiers from issue metadata, comments, ADF media refs, attachments, and linked URLs.
- The queue manager consumes these dossiers as untrusted grounding context and uses blocking gaps to downgrade write-capable tasks to read-only planning mode.

### Persistence Layer

- Runtime state, scheduler state, control-plane records, audit events, memory metadata, and knowledge governance persist through Postgres-first domain stores under [`../../koda/state`](../../koda/state), [`../../koda/services/runtime`](../../koda/services/runtime), and [`../../koda/knowledge/v2`](../../koda/knowledge/v2).
- Production runtime state is Postgres-first and object-backed; local disk is scratch-only and never the source of truth.
- Object-backed binaries are stored by logical `object_key`; local disk is scratch-only and must stay outside the repository.
- Per-agent runtime namespacing and scratch roots are defined in [`../../koda/config.py`](../../koda/config.py) and [`../../koda/control_plane/settings.py`](../../koda/control_plane/settings.py).

### Scheduler Layer

- [`../../koda/services/scheduled_jobs.py`](../../koda/services/scheduled_jobs.py) is the unified scheduler domain facade and persistence surface.
- [`../../koda/services/scheduled_job_dispatcher.py`](../../koda/services/scheduled_job_dispatcher.py) owns due-run materialization, catch-up, leasing, and dispatcher lifecycle.
- [`../../koda/services/scheduled_job_runtime.py`](../../koda/services/scheduled_job_runtime.py) owns scheduled execution dispatch, run transitions, verification, retries, and notifications.
- Scheduled agent jobs execute through the same queue/runtime path as normal Telegram requests, so provider fallback, canonical sessions, memory, retrieval, approvals, and verification remain consistent.

### Memory Layer

- [`../../koda/memory`](../../koda/memory) adds recall before a provider run and extraction after a response is produced.
- The memory layer is intentionally best-effort and should never block the core runtime path.

## External Dependencies And Integrations

- Telegram Bot API through `python-telegram-bot`
- Claude Code CLI through the `claude` command
- Codex CLI through the `codex` command
- Playwright for browser automation
- GitHub CLI, GitLab CLI, Docker, AWS CLI, and Google Workspace CLI for shell-driven integrations
- Jira and Confluence through the Atlassian client layer
- PDF, DOCX, spreadsheet, OCR, and media extraction libraries for artifact dossiers
- PostgreSQL through `asyncpg`, with optional SSL and SSH tunneling
- sentence-transformers for embeddings and pgvector-backed recall/search
- Whisper CLI and Kokoro or ElevenLabs for audio workflows

## Naming And Namespace Rules

- A `AGENT_ID` can be supplied at startup to namespace configuration and paths.
- Compiled prompt documents are namespaced by agent ID in the control plane and materialized into runtime state during publish/apply.
- Scratch directories are derived from the agent ID inside [`../../koda/config.py`](../../koda/config.py), but canonical state remains in Postgres/object storage.
- Repo-root artifacts such as local databases, smoke workspaces, and materialized scratch state are residue and should fail operational checks in the stateless rollout.
- Runtime sessions are canonical to the agent. Each provider only keeps a mapped native session or thread id for resume and fallback continuity.
